ЧАСТЬ 1

Как работает SSH соединение

Этапы установки соединения

Клиент (сервер A)                    Сервер (сервер B)
      |                                      |
      |-------- TCP SYN ------------------>  |   1. TCP handshake
      |<------- TCP SYN-ACK ---------------  |
      |-------- TCP ACK ------------------>  |
      |                                      |
      |<------- Banner (43 байта) ---------- |   2. Banner exchange
      |-------- Banner (43 байта) -------->  |      "SSH-2.0-OpenSSH_9.6p1"
      |                                      |
      |-------- KEXINIT (1000-1500 байт) --> |   3. Key exchange init
      |<------- KEXINIT ------------------- |      список алгоритмов
      |                                      |
      |<-------- KEX_ECDH_REPLY ----------- |   4. Обмен ключами
      |-------> NEWKEYS ------------------>  |
      |<------- NEWKEYS ------------------- |
      |                                      |
      |-------> AUTH_REQUEST ------------->  |   5. Аутентификация
      |<------- AUTH_SUCCESS -------------- |
      |                                      |
      |======= Зашифрованный туннель ======  |   6. Туннель работает ✓

SSH Banner

Первое что отправляет каждая сторона сразу после TCP соединения. Строка вида:

SSH-2.0-OpenSSH_9.6p1 Ubuntu-3ubuntu13.16
Размер: ~43 байта. Если banner не обменялся — соединение не пойдёт дальше.

KEXINIT (Key Exchange Init)

Большой пакет (~1000-1500 байт) содержащий список алгоритмов шифрования, MAC и случайный cookie.

⚠ Если KEXINIT не доходит — соединение зависает именно здесь. Самое частое место зависания при проблемах с MTU или фильтрацией.

Флаги TCP в tcpdump

ФлагЗначение
[S]SYN — инициация соединения
[S.]SYN-ACK — подтверждение соединения
[.]ACK — подтверждение получения данных
[P.]PSH+ACK — отправка данных
[F.]FIN — закрытие соединения
[R.]RST — сброс соединения

seq и ack в tcpdump

IP A > B: Flags [P.], seq 1:44, ack 1, length 43

seq 1:44  — A отправляет байты с 1 по 44 (т.е. 43 байта)
ack 1     — A подтверждает что получил от B байты до позиции 1 (ещё ничего)
length 43 — размер данных в пакете

Если ack не меняется долго — сторона не получает данные от собеседника.

MTU и MSS

MTU — максимальный размер пакета на интерфейсе (обычно 1500 байт)
MSS — максимальный размер TCP сегмента (обычно MTU - 40 = 1460 байт)

Если где-то в сети меньший MTU — большие пакеты фрагментируются или отбрасываются. KEXINIT как раз большой пакет.
ЧАСТЬ 2

Пошаговая диагностика

1
Проверить базовую доступность
▾
# Пингуется ли сервер B с сервера A?
ping -c 3 <IP_сервера_B>

# Открыт ли порт SSH?
nc -zv <IP_сервера_B> <порт>

# Трассировка маршрута
traceroute <IP_сервера_B>
Если nc говорит Connection refused — порт закрыт или sshd не слушает
Если nc говорит succeeded — порт открыт, идём дальше
2
Проверить что sshd слушает нужный порт
▾
# На сервере B
sudo ss -tlnp | grep ssh
sudo ss -tlnp | grep <порт>
LISTEN 0 128 0.0.0.0:22 ... sshd — хорошо
Если пусто — sshd не запущен или слушает другой порт.
3
Подключиться с verbose и смотреть где зависает
▾
ssh -v -p <порт> user@<IP>
Где завислоЧто значит
Connecting to...Порт недоступен, файрвол
Connection established затем тишинаBanner не получен от сервера
SSH2_MSG_KEXINIT sentKEXINIT отправлен, ждём ответ — проблема MTU или фильтрация
Authentications that can continue: publickeyКлюч не принят
Accepted publickey✓ Всё хорошо
4
Смотреть трафик через tcpdump на обоих серверах
▾
# На сервере A (клиент)
sudo tcpdump -i eth0 host <IP_сервера_B> -n

# На сервере B (сервер)
sudo tcpdump -i ens3 host <IP_сервера_A> -n

Нормальное соединение:

A > B: [S]           ← отправил SYN
B > A: [S.]          ← получил SYN-ACK
A > B: [.]           ← отправил ACK
B > A: [P.] length 43  ← получил banner от B  ✓
A > B: [P.] length 43  ← отправил banner
B > A: [.] ack 44    ← B подтвердил banner    ✓
A > B: [.] length 1148 ← отправил KEXINIT
B > A: [.] ack 1192  ← B подтвердил KEXINIT  ✓
B > A: [P.] length 1xxx ← получил KEXINIT от B ✓

Признаки проблем:

# Проблема 1 — banner не доходит до A
A > B: [S]
B > A: [S.]
A > B: [.]
# Дальше тишина

# Проблема 2 — KEXINIT не доходит до B
B > A: [P.] length 43   ← banner от B
A > B: [P.] length 43   ← banner от A
A > B: [.] length 1148  ← KEXINIT от A (большой пакет)
B > A: [P.] length 43   ← B продолжает слать banner, KEXINIT не получил!
5
Проверить логи sshd на сервере B
▾
# Смотреть в реальном времени
sudo journalctl -u ssh -f

# Или auth.log
sudo tail -f /var/log/auth.log
ЗаписьЗначение
Accepted publickey for userУспешный вход по ключу
banner exchange: invalid formatПодключается не SSH клиент (боты)
Connection closed by X port YСоединение закрыто до аутентификации
Failed password for userНеверный пароль
Invalid user XПользователь не существует
Если IP сервера A не появляется в логах — соединение обрывается до sshd (файрвол, fail2ban, hosts.deny).
6
Проверить файрвол и блокировщики
▾
# UFW
sudo ufw status verbose

# iptables — входящие
sudo iptables -L INPUT -n --line-numbers

# iptables — исходящие
sudo iptables -L OUTPUT -n

# NAT правила
sudo iptables -t nat -L -n

# Mangle (TPROXY, изменение пакетов)
sudo iptables -t mangle -L -n

# fail2ban
sudo fail2ban-client status sshd

# hosts.deny
cat /etc/hosts.deny
7
Проверить проблему MTU
▾
# Текущий MTU
ip link show | grep mtu

# Временно уменьшить MTU
sudo ip link set eth0 mtu 1200

# Ограничить MSS для SSH — на сервере B (исходящие)
sudo iptables -t mangle -A POSTROUTING -p tcp --sport 22 -j TCPMSS --set-mss 1100

# На сервере A (исходящие к серверу B)
sudo iptables -t mangle -A OUTPUT -p tcp --dport 22 -j TCPMSS --set-mss 1100
8
Проверить проходимость трафика напрямую
▾
# На сервере B — поднять listener
python3 -c "
import socket
s = socket.socket()
s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
s.bind(('0.0.0.0', 9999))
s.listen(1)
print('waiting...')
conn, addr = s.accept()
print(f'connected from {addr}')
data = conn.recv(4096)
print(f'received {len(data)} bytes')
conn.send(b'OK:' + data[:20])
conn.close()
"

# На сервере A — отправить данные
python3 -c "
import socket
s = socket.create_connection(('<IP_сервера_B>', 9999))
s.send(b'HELLO' * 300)
print('sent')
resp = s.recv(1000)
print('received:', resp)
"
⚠ Если TCP не работает, а UDP работает — провайдер фильтрует TCP после handshake.
Решение: WireGuard (работает по UDP). 
# Тест UDP — на сервере B
python3 -c "
import socket
s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
s.bind(('0.0.0.0', 9999))
data, addr = s.recvfrom(4096)
print(f'received {len(data)} bytes from {addr}')
"

# На сервере A
python3 -c "
import socket
s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
s.sendto(b'HELLO' * 300, ('<IP_сервера_B>', 9999))
print('sent')
"

Проверить проблему с ключами
▾
# Публичный ключ совпадает с приватным?
ssh-keygen -y -f ~/.ssh/tunnel_key
# Вывод должен совпадать с authorized_keys на сервере B

# Права на файлы (критично!)
ls -la ~/.ssh/
# authorized_keys должен быть 600, папка .ssh — 700

# Права на домашнюю папку
ls -la /home/ | grep <username>
ПроблемаСимптомРешение
Неверные права на ~/.sshПросит парольchmod 700 ~/.ssh
Неверные права на authorized_keysПросит парольchmod 600 ~/.ssh/authorized_keys
Папка принадлежит другому юзеруПросит парольchown user:user /home/user
Публичный ключ не тотПросит парольПересоздать и скопировать
10
Socket activation (Ubuntu специфика)
▾
На новых Ubuntu sshd запускается через systemd socket. Добавление Port в sshd_config не поможет — нужно отключить socket activation. 
# Проверить запущен ли через socket
systemctl status ssh | grep TriggeredBy

# Если есть "TriggeredBy: ssh.socket" — отключить:
sudo systemctl disable ssh.socket
sudo systemctl stop ssh.socket
sudo systemctl restart ssh.service

# Теперь sshd читает Port из sshd_config

4
UDP работает Значит решение через WireGuard или reverse tunnel
Вывод: когда TCP данные после handshake не доходят но UDP работает — это блокировка на уровне провайдера. Решение: WireGuard туннель (UDP) или обращение в поддержку провайдера.

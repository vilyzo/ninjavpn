Файловая система
Навигация, просмотр, создание, удаление файлов и директорий
ls
Список файлов и директорий. Самая частая команда — используется постоянно для навигации и аудита содержимого папок.
Примеры
$ ls                      # простой список
$ ls -la                  # детально + скрытые файлы
$ ls -lh /var/log/       # размеры в читаемом виде
$ ls -lt                  # сортировка по времени изменения
$ ls -lS                  # сортировка по размеру (большие сверху)
Ключи
-lдлинный формат
-aскрытые файлы
-hчитаемый размер
-tсортировка по дате
-Sсортировка по размеру
-Rрекурсивно
find
Поиск файлов по имени, размеру, времени изменения, владельцу и т.д. Незаменим для аудита и автоматизации. Помни: -exec ... \; вызывает команду для каждого файла, а -exec ... + группирует — в разы быстрее.
Примеры
$ find /etc -name "*.conf"          # найти все .conf файлы
$ find /var -size +100M            # файлы > 100MB
$ find . -mtime -1                  # изменённые за последние 24ч
$ find /tmp -type f -delete        # найти и удалить файлы в /tmp
$ find . -perm 777                  # файлы с правами 777
$ find /var/log -mtime +30 -exec rm {} + # удалить старше 30 дней (быстро)
$ find . -type f -name "*.log" -exec gzip {} \;  # сжать каждый
$ find . -empty                     # пустые файлы/папки
$ find . -user deploy -group www-data  # по владельцу и группе
$ find . -newer /tmp/marker         # изменённые после marker-файла
Ключи
-nameпо имени (glob)
-inameрегистронезависимо
-typef=файл, d=папка, l=link
-size+/-N[kMG]
-mtime/-atimeдни modify/access
-permпо правам
-user/-groupвладелец
-exec ... +группа аргументов
-deleteудалить найденное
-maxdepth Nглубина поиска
-pruneисключить ветку
cp / mv / rm
Базовые операции с файлами: копирование, перемещение/переименование, удаление. Часто используются в скриптах автоматизации.
Примеры
$ cp -r /src/ /dst/             # рекурсивное копирование
$ cp -p file.conf file.conf.bak  # копия с сохранением прав
$ mv oldname newname            # переименование
$ rm -rf /tmp/cache/           # удалить рекурсивно без подтверждения
$ rm -i *.log                    # с подтверждением каждого файла
Ключи (cp/rm)
-rрекурсивно
-fбез подтверждения
-iинтерактивно
-pсохранить права/дату
-vverbose вывод
ln
Создание ссылок. Hard link — это второе имя для того же inode (нельзя через FS-границы и на директории). Symlink — отдельный файл-указатель на путь.
Примеры
$ ln file hardlink                  # hard link
$ ln -s /opt/app-v1.2 /opt/app    # symlink (классика deploy)
$ ln -sfn /opt/app-v1.3 /opt/app  # атомарно обновить symlink (zero-downtime)
$ readlink -f /opt/app             # куда указывает (абсолютный путь)
$ stat file                          # inode, время, права, size
Ключи
-ssymbolic link
-fперезаписать существующий
-nне разыменовывать target
xargs
Превращает stdin в аргументы команды. Связка find ... | xargs ... мощнее -exec, особенно с -P (параллелизм).
Примеры
$ find . -name "*.log" | xargs rm     # НЕ безопасно с пробелами в именах
$ find . -name "*.log" -print0 | xargs -0 rm  # безопасно (NUL-разделитель)
$ cat urls.txt | xargs -n 1 -P 8 curl -sO     # 8 параллельных загрузок
$ echo "1 2 3" | xargs -I {} echo "PID={}"     # placeholder {}
$ ls *.bak | xargs -p rm                # спросить подтверждение
Ключи
-0NUL-разделитель
-n NN аргументов за раз
-P NN параллельно
-I {}placeholder
-pподтверждение
-tverbose (echo cmd)
file / which / type
Идентификация файлов и команд. file определяет тип по magic-байтам (не по расширению). which — путь к бинарю в $PATH. type — alias/builtin/file.
Примеры
$ file mystery.bin                   # ELF / ASCII / image / ...
$ which docker                       # /usr/bin/docker
$ type ll                            # ll is aliased to 'ls -la'
$ command -v docker                  # POSIX-вариант (для скриптов)
💡 На собеседовании часто спрашивают разницу между hard link и symlink. Hard link — другое имя для того же inode. Symlink — указатель на путь. Создаются через ln file hardlink и ln -s file symlink.
🎤 Кейс на интервью: "df говорит, что диск полный, но du показывает мало места — что делать?" Ответ: 1) проверь inode-исчерпание df -i; 2) ищи удалённые, но открытые файлы lsof | grep deleted (нужен restart процесса или : > /proc/PID/fd/N); 3) скрытые mount points поверх — du -x с --one-file-system.

Процессы и мониторинг
Просмотр, управление и отладка процессов в системе
ps
Снимок текущих процессов. В связке с grep — мощный инструмент поиска. RSS — реально занятая RAM (KB), VSZ — виртуальная (включая swap и mmap). STAT: R=running, S=sleeping, D=uninterruptible (часто IO-wait), Z=zombie, T=stopped.
Примеры
$ ps aux                          # все процессы всех пользователей
$ ps aux | grep nginx             # найти процессы nginx
$ ps -ef --forest                # дерево процессов
$ ps -p 1234 -o pid,ppid,cmd,%cpu,%mem  # инфо о конкретном PID
$ ps aux --sort=-%mem | head      # топ-10 по памяти
$ ps aux --sort=-%cpu | head      # топ-10 по CPU
$ ps -eo pid,user,stat,wchan,cmd | grep " D "  # процессы в uninterruptible sleep
$ pgrep -fa nginx                    # PIDs + cmd по имени
Ключи
aвсе пользователи
uuser-ориентированный
xбез tty
-eвсе процессы
--forestдерево
-oформат вывода
--sortсортировка
top / htop
Интерактивный мониторинг системы в реальном времени. htop — улучшенная версия с цветным интерфейсом и удобным управлением.
Примеры
$ top                          # запустить мониторинг
$ top -u www-data             # только процессы пользователя
$ top -b -n 1 | head -20     # batch-режим (для скриптов)
$ htop -d 5                    # обновление каждые 0.5 сек
  # В top: P — сорт. по CPU, M — по памяти, k — kill, q — выход
Ключи top
-uфильтр по юзеру
-bbatch режим
-nчисло итераций
-dинтервал (деситые)
kill / pkill
Отправка сигналов процессам. SIGTERM (15) — вежливое завершение. SIGKILL (9) — принудительное. pkill позволяет убивать по имени.
Примеры
$ kill -15 1234            # SIGTERM — дать шанс завершиться
$ kill -9 1234             # SIGKILL — убить немедленно
$ pkill -f "python app.py"  # убить по полному имени команды
$ kill -HUP $(cat /var/run/nginx.pid)  # перечитать конфиг nginx
$ killall -u baduser        # убить все процессы пользователя
Сигналы
-1 HUPперечитать конфиг
-2 INTпрервать (Ctrl+C)
-9 KILLпринудительно
-15 TERMмягкое завершение
nice / renice / ionice
Управление приоритетом процессов. nice-уровень от -20 (макс. приоритет) до +19 (минимум). Только root может понижать (давать больше CPU). ionice — приоритет I/O, критично для бэкапов на загруженной системе.
Примеры
$ nice -n 10 ./backup.sh             # запустить с пониженным приоритетом
$ renice -n 5 -p 1234               # изменить приоритет работающего
$ renice -n 10 -u user1              # все процессы пользователя
$ ionice -c 3 rsync /data/ /backup/   # idle I/O — не мешает другим
$ ionice -c 2 -n 7 -p 1234           # best-effort, lowest priority
$ chrt -f 99 -p 1234                  # real-time приоритет (осторожно!)
ionice классы
-c 1real-time
-c 2best-effort (default)
-c 3idle (только если свободно)
-n 0-7priority внутри класса
nohup / disown / & / screen / tmux
Запуск процессов в фоне с отвязкой от терминала. & — в фон, nohup — игнорировать SIGHUP при выходе из ssh. Для интерактивных длинных сессий — tmux/screen.
Примеры
$ ./long-task.sh &                  # в фон (текущая сессия)
$ nohup ./long-task.sh > out.log 2>&1 &  # переживёт logout
$ jobs                              # фоновые задачи в текущем shell
$ fg %1                             # вернуть на передний план
$ disown -h %1                       # отвязать от shell
$ tmux new -s deploy                  # tmux-сессия
$ tmux attach -t deploy               # вернуться к сессии
  # tmux: Ctrl+b d — detach, Ctrl+b " — split, Ctrl+b c — new window
Сигнал№Что делаетМожно перехватить?
SIGHUP1Hangup. Часто = "перечитай конфиг" (nginx, sshd)Да
SIGINT2Прерывание (Ctrl+C)Да
SIGQUIT3Quit + core dumpДа
SIGKILL9Принудительное убийство — мгновенноНет
SIGUSR1/210/12Пользовательские (для своей логики)Да
SIGTERM15Запрос на завершение (вежливое)Да
SIGSTOP19Приостановить процессНет
SIGCONT18Продолжить остановленныйДа
💡 На интервью: всегда сначала пробуй SIGTERM (-15), и только если процесс не реагирует — SIGKILL (-9). SIGKILL нельзя перехватить или игнорировать.
🎤 Кейс: "У процесса статус D (uninterruptible sleep), kill -9 не помогает — почему?" Ответ: процесс ждёт I/O в ядре (обычно NFS, повисший диск или kernel bug). SIGKILL доставит сигнал только когда process выйдет из syscall. Решения: дождаться timeout, перемонтировать NFS, ребут. На зомби (Z) kill тоже не действует — нужен kill родителя.

Сеть и диагностика
Подключения, маршруты, порты, трафик
ss / netstat
Просмотр активных сетевых соединений и открытых портов. ss — современный быстрый аналог netstat (читает из /proc/net напрямую). Критично для отладки сервисов и поиска кто слушает порт.
Примеры
$ ss -tulnp                 # все слушающие порты с процессами
$ ss -an | grep :80        # соединения на порт 80
$ ss -s                     # статистика соединений
$ ss -tn state established   # только установленные TCP
$ ss -tn state time-wait | wc -l  # сколько в TIME_WAIT
$ ss -tnp '( dport = :443 or sport = :443 )'  # фильтр по порту
$ netstat -tulnp            # аналог через netstat (старый способ)
Ключи ss
-tTCP
-uUDP
-lтолько слушающие
-nчисла без DNS
-pпоказать процесс
-sстатистика
-xunix sockets
-4/-6IPv4/IPv6
curl
HTTP-клиент для тестирования API, скачивания файлов, диагностики веб-сервисов. Один из главных инструментов DevOps-инженера.
Примеры
$ curl -I https://example.com           # только заголовки ответа
$ curl -X POST -H "Content-Type: application/json" \
       -d '{"key":"val"}' https://api/endpoint  # POST JSON
$ curl -u user:pass https://secure.api    # Basic Auth
$ curl -o file.tar.gz -L https://url    # скачать с редиректом
$ curl -w "%{http_code}" -s -o /dev/null https://site  # только код ответа
Ключи
-Iтолько заголовки
-Xметод (POST/PUT/...)
-Hзаголовок
-dтело запроса
-oфайл вывода
-Lследовать редиректу
-ssilent mode
-kигнорировать SSL
ping / dig / host
Базовая диагностика. ping — ICMP-доступность (но многие узлы её блокируют). dig — DNS-отладка. host — короткий вывод для скриптов.
Примеры
$ ping -c 4 8.8.8.8                # 4 пакета к DNS Google
$ dig google.com A                  # A-запись (IPv4)
$ dig +short google.com            # только результат
$ dig @8.8.8.8 example.com MX     # MX через конкретный DNS
$ dig +trace example.com           # полная цепочка DNS-серверов
$ dig -x 8.8.8.8                   # reverse lookup (PTR)
$ host -t TXT example.com            # TXT-записи (SPF, DMARC, верификации)
Ключи dig
+shortкраткий ответ
+traceтрассировка DNS
@serverDNS-сервер
A/AAAA/MX/CNAME/TXTтип записи
-xreverse
nc (netcat) / nmap
"Швейцарский нож" сетей. nc — проверка портов, простейший chat, передача файлов. nmap — серьёзное сканирование. На проде: используй для проверки firewall и доступности сервиса.
Примеры
$ nc -zv example.com 443           # проверить открыт ли порт
$ nc -zv 10.0.0.5 20-25             # диапазон портов
$ nc -l 8080                       # слушать порт (для теста firewall)
$ nc -l 9000 > received.tar        # принять файл
$ nc target.host 9000 < file.tar    # отправить файл
$ nmap -sT -p 80,443,22 10.0.0.0/24  # TCP-connect scan подсети
$ nmap -sV -p 22 host                # версия сервиса
mtr / traceroute
mtr = traceroute + ping в реальном времени, показывает loss на каждом хопе. Лучший инструмент для диагностики "у нас тормозит сеть".
Примеры
$ mtr google.com                     # интерактивно
$ mtr -rwbz -c 100 google.com         # отчёт: 100 пакетов, no DNS, ASN
$ mtr --tcp -P 443 host                # TCP-трассировка к 443 (через NAT/FW)
$ traceroute -T -p 443 host           # TCP traceroute
TCP StateЧто означаетНа что обратить внимание
LISTENСокет ждёт входящих соединенийНорма для сервера
ESTABLISHEDАктивное соединение, идёт обмен даннымиНорма
SYN-SENTОтправлен SYN, ждём SYN-ACKМного = клиент не достучится
TIME-WAITЗакрыли мы; ждём 2*MSL (~60c)Тысячи = высокая частота кратких соединений, нужен keep-alive
CLOSE-WAITУдалённый закрыл, мы — нетУтечка в приложении (не закрывает сокеты)
FIN-WAIT-1/2Мы закрыли, ждём ACK / FINНорма; долгий FIN-WAIT-2 → сервер не отвечает
LAST-ACK / CLOSINGПереходные при закрытииКратковременны
🎤 Кейс: "Сервис не отвечает на порту 8080 — куда смотреть?" Чеклист: 1) ss -tlnp | grep 8080 — слушает ли вообще; 2) если слушает — curl -v localhost:8080 с локалхоста; 3) firewall: iptables -L -n / ufw status; 4) nc -zv host 8080 снаружи; 5) tcpdump -i any port 8080 — приходят ли пакеты; 6) Security Group / NACL в облаке.

Текст и работа с логами
grep, sed, awk, tail — основа анализа логов в production
grep
Поиск по шаблону (regex) внутри файлов или потоков. Ключевой инструмент анализа логов и конфигов на продакшне.
Примеры
$ grep -i "error" /var/log/nginx/error.log   # без учёта регистра
$ grep -r "password" /etc/               # рекурсивно по папке
$ grep -v "DEBUG" app.log               # инвертировать (исключить)
$ grep -E "ERROR|WARN|CRITICAL" app.log # расширенный regex
$ grep -c "500" access.log             # подсчёт совпадений
$ grep -A 3 -B 2 "exception" app.log  # контекст: 3 строки после, 2 до
Ключи
-iбез учёта регистра
-rрекурсивно
-vинвертировать
-Eextended regex
-cподсчёт строк
-nномер строки
-A/-Bконтекст после/до
-lтолько имена файлов
awk
Язык обработки текста по столбцам. Незаменим для парсинга структурированных логов, CSV-файлов и вывода команд.
Примеры
$ awk '{print $1, $4}' access.log       # вывести 1-й и 4-й столбцы
$ awk -F: '{print $1}' /etc/passwd     # разделитель ':' → имена юзеров
$ awk '$9 == "500"' access.log          # строки где поле 9 = 500
$ awk '{sum+=$5} END {print sum}' data  # сумма 5-го столбца
$ ps aux | awk '{print $2, $3}' | sort -k2 -rn | head  # топ по CPU
Ключи
-Fразделитель полей
-vпеременная
NRномер строки
NFчисло полей
BEGIN/ENDдо/после обработки
sed
Stream EDitor. Главное применение — замена/удаление строк в конфигах и логах. -i правит файл на месте (обязательно с бэкапом!). Поддерживает regex и адресацию по строкам.
Примеры
$ sed 's/foo/bar/g' file              # замена в выводе
$ sed -i.bak 's/foo/bar/g' file      # in-place + бэкап file.bak
$ sed -i '/^#/d' config.conf          # удалить комментарии
$ sed -i '/^$/d' file                  # удалить пустые строки
$ sed -n '10,20p' file                 # только строки 10–20 (как head/tail)
$ sed -i 's|/old/path|/new/path|g' file  # другой разделитель для путей
$ sed -i '/^bind-address/c\bind-address = 0.0.0.0' my.cnf  # заменить строку целиком
Ключи и команды
-i[ext]in-place [+бэкап]
-nподавить вывод
-Eextended regex
s/x/y/gsubstitute global
ddelete
pprint
sort / uniq / wc
Базовая аналитика логов. Важно: uniq убирает только соседние дубли, поэтому почти всегда идёт после sort. Связка sort | uniq -c | sort -rn — стандартный приём топ-аналитики.
Примеры
$ sort file.txt | uniq                # уникальные строки
$ sort file | uniq -c | sort -rn      # подсчёт + топ
$ awk '{print $1}' access.log | sort | uniq -c | sort -rn | head  # топ-IP
$ wc -l file                         # количество строк
$ sort -k2 -rn data.txt                # сорт по 2-му полю, числам, убыв.
$ sort -u -t, -k3 data.csv             # уникальные по 3-му полю CSV
Ключи sort
-nчисловая сортировка
-rубывание
-k Nпо полю N
-tразделитель
-uуникальные (=uniq)
-hhuman (1K/1M/1G)
cut / tr / paste
Лёгкие текстовые трансформации. cut — вырезать поля/символы. tr — замена/удаление символов. paste — склейка строк по горизонтали.
Примеры
$ cut -d: -f1 /etc/passwd           # имена пользователей
$ cut -d',' -f2,4 data.csv           # 2-й и 4-й столбец CSV
$ echo "Hello" | tr '[:lower:]' '[:upper:]'  # HELLO
$ tr -d '\r' < win.txt > unix.txt      # убрать CR (CRLF → LF)
$ tr -s ' ' < file                    # squeeze повторяющиеся пробелы
$ paste file1 file2                    # построчная склейка через таб
tail / head / less
Чтение файлов. tail -f — мастхэв для мониторинга логов в реальном времени. less — постраничный просмотр больших файлов с поиском.
Примеры
$ tail -f /var/log/syslog             # следить за логом в реальном времени
$ tail -n 100 app.log | grep ERROR    # последние 100 строк + фильтр
$ tail -F app.log | grep --line-buffered "ERROR"  # live grep с reopen
$ tail -f a.log b.log                  # несколько файлов разом
$ head -20 config.yaml               # первые 20 строк
$ less +G app.log                    # открыть в конце файла
  # less: /pattern — поиск, n — следующее, G — в конец, g — в начало, &ERROR — фильтр
Ключи tail
-fследить за файлом
-nчисло строк
-Fследить + reopen при ротации
-cпо байтам
💡 Святая троица аналитики логов: grep | awk | sort | uniq -c | sort -rn | head. Топ-10 IP по запросам, топ-10 URL по 500-кам, топ user-agent — всё этой связкой.
🎤 Кейс — анализ access.log за день:
• Топ-10 IP: awk '{print $1}' access.log | sort | uniq -c | sort -rn | head
• 5xx за последний час: awk '$9 ~ /^5/' access.log | tail -1000
• RPS по минутам: awk '{print $4}' access.log | cut -d: -f1-3 | uniq -c
• Тяжёлые ответы: awk '$10 > 1048576 {print $7, $10}' access.log | sort -k2 -rn | head

Права, пользователи, sudo
Безопасность, управление доступом, разрешения на файлы
chmod / chown
Управление правами доступа и владельцем файлов. Критично для безопасности: 777 на продакшне = красный флаг на интервью.
Примеры
$ chmod 755 script.sh              # rwxr-xr-x (типично для скриптов)
$ chmod 644 config.conf            # rw-r--r-- (конфиги)
$ chmod -R 750 /var/app/          # рекурсивно на директорию
$ chmod u+x,g-w script.sh          # символьный режим
$ chown www-data:www-data -R /var/www/  # владелец:группа рекурсивно
Права (числа)
4read (r)
2write (w)
1execute (x)
7=rwxполный доступ
644файлы
755директории/скрипты
sudo / su / useradd
Управление привилегиями и пользователями. sudo — выполнить команду от root. su — сменить пользователя. useradd — создать нового.
Примеры
$ sudo systemctl restart nginx          # выполнить как root
$ sudo -u postgres psql               # выполнить как другой пользователь
$ sudo -l                              # что мне разрешено через sudo
$ useradd -m -s /bin/bash deploy      # создать пользователя с home
$ usermod -aG docker $USER           # добавить в группу docker
$ passwd deploy                        # задать пароль
Ключи useradd/usermod
-mсоздать home
-sshell пользователя
-Gдобавить в группы
-aGappend к группам
umask / SUID / SGID / Sticky
Дополнительные биты прав. umask — маска для новых файлов (вычитается из 666/777). SUID (4xxx) — выполнить от владельца файла (классика: /usr/bin/passwd). SGID (2xxx) — от группы или новые файлы наследуют группу директории. Sticky (1xxx) — удалять может только владелец (классика: /tmp).
Примеры
$ umask                              # посмотреть текущую (обычно 0022)
$ umask 0027                         # новые файлы: 640, папки: 750
$ chmod 4755 /usr/bin/myutil          # SUID + 755
$ chmod g+s /shared/dir               # SGID на папку (наследование группы)
$ chmod +t /tmp                       # sticky bit
$ find / -perm -4000 -type f 2>/dev/null   # все SUID — security аудит
setfacl / getfacl (ACL)
Расширенные права доступа (POSIX ACL) — когда обычных user/group/other не хватает. Нужны для случаев "дать конкретному пользователю доступ к чужому файлу" без изменения владельца. Признак: + в конце прав в ls -l.
Примеры
$ getfacl file                       # посмотреть ACL
$ setfacl -m u:deploy:rx /var/log/app/  # дать deploy r+x
$ setfacl -m g:devs:rw file           # группе devs — rw
$ setfacl -d -m u:deploy:rx /var/log/app/  # default ACL (новые файлы наследуют)
$ setfacl -x u:deploy file             # удалить запись
capabilities / getcap / setcap
Современная альтернатива SUID. Вместо "всё или ничего" даёт процессу конкретные права (например, биндить порт <1024). Безопаснее, чем root.
Примеры
$ getcap /usr/bin/ping               # посмотреть caps
$ setcap 'cap_net_bind_service=+ep' /usr/bin/node  # node может слушать 80/443 без root
$ setcap -r /usr/bin/node              # убрать caps
$ getpcaps $$                          # caps текущего процесса
  # Полезные: CAP_NET_BIND_SERVICE, CAP_SYS_PTRACE, CAP_DAC_READ_SEARCH, CAP_NET_ADMIN
💡 Числа прав легко: r=4, w=2, x=1. 755 = rwx r-x r-x (для исполняемых), 644 = rw- r-- r-- (для конфигов), 600 = rw- --- --- (для ключей и секретов).
🎤 Кейс: "Юзер в группе docker, но docker ps говорит permission denied — почему?" Группы добавляются в сессию при логине, текущая bash не знает. Решения: newgrp docker, su - $USER, или перелогиниться. Проверить: id (новые группы) vs groups (сессии).

Диск и системные ресурсы
Мониторинг дисков, памяти, нагрузки на систему
df / du
df — свободное место на разделах. du — сколько занимает конкретная директория. Первое, что проверяют при проблемах на продакшне.
Примеры
$ df -h                              # место на всех разделах
$ df -h /var                        # конкретный раздел
$ du -sh /var/log/                 # размер папки логов
$ du -h --max-depth=1 /var/      # топ папок в /var
$ du -sh * | sort -rh | head -10  # топ-10 папок по размеру
Ключи
-hhuman readable
-sтолько итог (du)
--max-depthглубина рекурсии
-iinodes (df)
free / vmstat / uptime
Мониторинг оперативной памяти и нагрузки. free — RAM/swap. vmstat — виртуальная память и I/O. uptime — LA (load average).
Примеры
$ free -h                 # RAM и swap в читаемом виде
$ vmstat 1 5             # 5 замеров каждую секунду
$ uptime                  # LA за 1, 5, 15 минут
$ cat /proc/meminfo       # детальная инфо о памяти
  # LA > кол-во CPU = перегрузка системы
vmstat столбцы
us/syuser/system CPU %
si/soswap in/out
bi/bodisk I/O блоки
waожидание I/O
lsblk / blkid / mount / fstab
Управление блочными устройствами и монтированием. lsblk — дерево дисков и партиций. blkid — UUID и тип FS. Монтирования прописываются в /etc/fstab (только UUID, не /dev/sdX — переименуются!).
Примеры
$ lsblk                              # все диски и партиции
$ lsblk -f                           # + файловая система и UUID
$ blkid                              # UUID всех устройств
$ mount | column -t                  # текущие монтирования
$ mount -o remount,ro /mnt/data      # перемонтировать как read-only
$ mount -t nfs server:/exp /mnt      # NFS
$ umount /mnt/data                  # отмонтировать (busy? — lsof, fuser)
$ mount -a                           # перечитать fstab (после правки)
$ findmnt /var                       # подробно про конкретную точку
swap / OOM killer
Swap — расширение RAM на диске (медленно). vm.swappiness (0–100) — насколько охотно ядро уходит в swap. Для серверов БД обычно ставят 10 или 1. OOM killer убивает процессы при нехватке памяти — обычно с худшим oom_score.
Примеры
$ swapon --show                       # активный swap
$ fallocate -l 2G /swapfile && chmod 600 /swapfile
$ mkswap /swapfile && swapon /swapfile  # создать и включить swap-файл
$ sysctl vm.swappiness=10              # временно
$ dmesg -T | grep -i "killed process"  # кого убил OOM
$ cat /proc/1234/oom_score             # шанс быть убитым
$ echo -1000 > /proc/1234/oom_score_adj # защитить от OOM
🎤 Кейс — high load average при низком CPU: LA включает процессы в R и D (uninterruptible sleep, обычно IO-wait). Если CPU свободен, а LA высокий — узкое место в I/O. Проверь: vmstat 1 (колонка wa), iostat -xz 1 (%util близко к 100), iotop (кто грузит диск).

Systemd, SSH, архивы
Управление сервисами, удалённый доступ, работа с архивами
systemctl
Управление systemd-сервисами. На каждом современном Linux-сервере это основной инструмент управления демонами. Свои unit-файлы кладутся в /etc/systemd/system/, после правки — обязательно systemctl daemon-reload.
Примеры
$ systemctl status nginx                # состояние сервиса
$ systemctl start|stop|restart nginx   # управление сервисом
$ systemctl reload nginx                # SIGHUP без рестарта (если поддерживается)
$ systemctl enable --now nginx          # автозапуск + старт сразу
$ systemctl list-units --type=service --state=failed  # упавшие
$ systemctl cat nginx                   # показать unit-файл
$ systemctl edit nginx                  # override в /etc/systemd/system/nginx.service.d/
$ systemctl daemon-reload               # после правки unit-файлов
$ systemd-analyze blame                 # что тормозит загрузку системы
$ journalctl -u nginx -f             # логи сервиса в реальном времени
$ journalctl -u nginx --since "1 hour ago"    # за последний час
Частые команды

reloadперечитать конфиг
enableавтозапуск ON
disableавтозапуск OFF
maskзапретить запуск
editoverride unit
daemon-reloadпосле правки unit
📄 Свой unit-файл
Минимальный unit для приложения. Type=simple — процесс не демонизируется. Restart=always — рестарт при падении. Безопасность: запускать от отдельного пользователя, не root.
/etc/systemd/system/myapp.service
[Unit]
Description=My App
After=network.target postgresql.service
Requires=postgresql.service

[Service]
Type=simple
User=appuser
Group=appuser
WorkingDirectory=/opt/myapp
EnvironmentFile=/etc/myapp/env
ExecStart=/opt/myapp/bin/server
ExecReload=/bin/kill -HUP $MAINPID
Restart=always
RestartSec=5
LimitNOFILE=65536
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
ssh / scp / rsync
Удалённый доступ и передача файлов. rsync мощнее scp: синхронизирует только дельту, поддерживает resume, сохраняет права. SSH-туннели — -L локальный (порт у тебя), -R reverse (порт на сервере), -D SOCKS5-прокси, -J jump host.
Примеры
$ ssh -i ~/.ssh/key.pem user@server       # подключение с ключом
$ ssh -L 8080:localhost:80 user@server    # локальный туннель → порт 80 удалённо
$ ssh -L 5432:db.internal:5432 bastion     # доступ к БД через бастион
$ ssh -R 9000:localhost:3000 user@server   # reverse: сервер:9000 → мой:3000
$ ssh -D 1080 user@server                # SOCKS5 прокси на 1080
$ ssh -J bastion@jump.host user@private     # jump через бастион
$ ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server  # залить ключ
$ ssh-keygen -t ed25519 -C "deploy@2026"     # создать ключ (ed25519 быстрее RSA)
$ scp file.tar.gz user@server:/tmp/      # скопировать файл на сервер
$ rsync -avz --delete /src/ user@srv:/dst/  # синхронизация
$ rsync -avz --exclude='node_modules' --dry-run /src/ srv:/dst/  # тест
$ rsync -avz -e "ssh -i key.pem" /local/ srv:/remote/
Ключи rsync
-aarchive (права, дата, симлинки)
-vverbose
-zсжатие
--deleteудалить лишнее на dst
--dry-runсимуляция без изменений
--excludeисключить шаблон
--progressпоказывать прогресс

  # теперь: ssh prod-db — и сразу туннель на БД через бастион
tar / gzip
Архивирование и сжатие. tar сам по себе не сжимает — только упаковывает. Сжатие добавляют флагами -z (gzip), -j (bzip2), -J (xz).

Pipes, редиректы, потоки
stdin/stdout/stderr, pipes, here-doc, process substitution — фундамент shell
stdin / stdout / stderr
Три стандартных потока: 0 stdin (ввод), 1 stdout (нормальный вывод), 2 stderr (ошибки). Разделение позволяет фильтровать только нужное и логировать ошибки отдельно.
Примеры
$ cmd > out.log                       # stdout → файл (перезаписать)
$ cmd >> out.log                      # stdout → файл (добавить)
$ cmd 2> err.log                      # stderr → файл
$ cmd > out.log 2>&1                  # stderr → туда же, куда stdout
$ cmd &> combined.log                 # bash-shortcut для предыдущего
$ cmd 2>/dev/null                     # подавить ошибки
$ cmd >/dev/null 2>&1                 # подавить вообще всё
$ cmd < input.txt                     # файл → stdin
$ cmd1 | cmd2 |& cmd3                 # pipe stdout И stderr (bash 4+)
Шпаргалка
>stdout overwrite
>>stdout append
2>stderr
2>&1stderr → stdout
&>оба в файл
/dev/nullмусорка
tee
Раздвоить вывод: одновременно в файл и на экран. Незаменимо для долгих команд, когда хочешь и наблюдать, и сохранить лог. С sudo — единственный нормальный способ записать в защищённый файл из пайпа.
Примеры
$ make install 2>&1 | tee build.log      # смотрим и сохраняем
$ cmd | tee -a log.txt                  # append
$ echo "127.0.0.1 myhost" | sudo tee -a /etc/hosts  # sudo + редирект
$ cmd | tee >(grep ERROR > err.log) | wc -l   # process substitution
here-doc / here-string
Передать многострочный текст команде через stdin. Часто используется в скриптах для конфигов, SQL-запросов, SSH-сессий с командами.
Примеры
$ cat <<EOF > config.yaml
server:
  port: 8080
EOF

$ ssh server <<'EOF'
  cd /opt/app
  git pull
  systemctl restart app
EOF                                          # 'EOF' — без подстановки переменных

$ psql <<-EOF
  SELECT count(*) FROM users;
EOF                                          # <<- убирает leading tabs

$ grep "error" <<< "$LOG_LINE"          # here-string (одна строка)
Цепочка операторов
Контроль выполнения по результату. $? — код возврата последней команды (0 = успех).
Примеры
$ make build && make test             # test только если build успешен
$ curl -sf $URL || echo "FAILED"       # echo только при ошибке
$ cmd1; cmd2; cmd3                     # последовательно (независимо от exit code)
$ (cd /tmp && ls)                      # subshell — cd не повлияет на текущий
$ { echo A; echo B; } > file           # группировка без subshell
$ cmd; echo "exit=$?"                  # посмотреть код возврата
$ set -euo pipefail                   # строгий режим для скриптов

-uunset variable = error

-xtrace команд

Cron и systemd timers
Запуск задач по расписанию: бэкапы, ротация, очистка
crontab
Классика планировщика. Формат: m h dom mon dow. Каждое поле принимает число, *, диапазон 1-5, список 1,15 или шаг */5. Важно: cron не наследует $PATH из shell — указывай абсолютные пути.
Примеры
$ crontab -e                          # редактировать свой crontab
$ crontab -l                          # посмотреть
$ crontab -u deploy -l                 # crontab другого пользователя
  # формат строки:
  # ┌─ минута (0-59)
  # │  ┌─ час (0-23)
  # │  │  ┌─ день месяца (1-31)
  # │  │  │  ┌─ месяц (1-12)
  # │  │  │  │  ┌─ день недели (0-7, 0 и 7 = вс)
  # │  │  │  │  │
  # *  *  *  *  * command

0 3 * * *      /opt/backup.sh >> /var/log/backup.log 2>&1   # каждый день в 3:00
*/5 * * * *    /usr/local/bin/healthcheck.sh                # каждые 5 минут
0 */6 * * *    /opt/sync.sh                                 # каждые 6 часов
0 0 * * 0      /opt/weekly-cleanup.sh                       # каждое воскресенье
@reboot        /opt/start.sh                                # при старте
@daily         /opt/backup.sh                               # раз в сутки (= 0 0 * * *)
Где смотреть
/etc/crontabсистемный
/etc/cron.d/drop-in файлы
/var/spool/cron/пользовательские
/var/log/cronлоги
systemd timer
Современная альтернатива cron. Плюсы: ведёт логи через journalctl, поддерживает OnCalendar в человеческом формате, может запускать пропущенные при выключении (Persistent=true), интегрирован с systemd-сервисами.
backup.timer + backup.service
# /etc/systemd/system/backup.service
[Unit]
Description=Daily backup

[Service]
Type=oneshot
ExecStart=/opt/backup.sh

# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily

[Timer]
OnCalendar=*-*-* 03:00:00
RandomizedDelaySec=10min
Persistent=true

[Install]
WantedBy=timers.target

$ systemctl enable --now backup.timer      # активация
$ systemctl list-timers                  # все таймеры: next run, last run
$ journalctl -u backup.service          # логи выполнения
logrotate
Ротация логов: сжимает старые, удаляет очень старые, не даёт диску забиться. Запускается через cron.daily или systemd-timer. Конфиги в /etc/logrotate.d/.
/etc/logrotate.d/myapp
/var/log/myapp/*.log {
    daily
    rotate 14
    compress
    delaycompress
    missingok
    notifempty
    create 0640 myapp myapp
    sharedscripts
    postrotate
        systemctl reload myapp > /dev/null 2>&1 || true
    endscript
}

$ logrotate -d /etc/logrotate.d/myapp     # debug (dry-run)
$ logrotate -f /etc/logrotate.conf          # принудительная ротация
💡 Главные грабли cron: 1) $PATH минимальный — используй абсолютные пути; 2) вывод и stderr пропадают, если не редиректить (>> file 2>&1); 3) процент % надо экранировать \%; 4) проверь часовой пояс сервера (timedatectl).

Безопасность и Firewall
iptables, nftables, ufw, fail2ban, аудит
iptables / nftables
Netfilter — firewall в ядре Linux. iptables — классика, nftables — современная замена (один tool вместо iptables/ip6tables/arptables). Правила обрабатываются сверху вниз, первое совпадение выигрывает.
Примеры iptables
$ iptables -L -n -v --line-numbers     # все правила с счётчиками
$ iptables -A INPUT -p tcp --dport 22 -j ACCEPT     # разрешить SSH
$ iptables -A INPUT -s 10.0.0.0/8 -j ACCEPT     # разрешить подсеть
$ iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
$ iptables -A INPUT -j DROP               # дропать остальное (последнее правило!)
$ iptables -D INPUT 5                    # удалить правило №5
$ iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8080
$ iptables-save > /etc/iptables/rules.v4  # сохранить (правила сбросятся после ребута!)

ufw / firewalld
Обёртки над iptables/nftables для людей. ufw (Ubuntu) — простая. firewalld (RHEL/CentOS) — с концепцией зон.
Примеры
$ ufw status verbose                  # текущее состояние
$ ufw default deny incoming            # по умолчанию запретить
$ ufw allow 22/tcp                     # SSH
$ ufw allow from 10.0.0.0/8 to any port 5432  # БД только из VPC
$ ufw limit 22/tcp                     # rate-limit (защита от brute-force)
$ ufw enable                           # активировать

$ firewall-cmd --list-all             # firewalld: показать всё
$ firewall-cmd --permanent --add-service=https
$ firewall-cmd --reload                # применить изменения
SSH hardening / fail2ban
Минимальный security-чеклист для production-сервера. SSH — это первая дверь, через неё ломаются чаще всего.
/etc/ssh/sshd_config
PermitRootLogin no                  # никогда root по ssh
PasswordAuthentication no           # только ключи
PubkeyAuthentication yes
ChallengeResponseAuthentication no
UsePAM yes
X11Forwarding no
MaxAuthTries 3
ClientAliveInterval 300
AllowUsers deploy ops              # whitelist
Port 2222                           # не панацея, но снижает шум в логах

$ sshd -t                              # проверить конфиг ПЕРЕД рестартом!
$ systemctl restart sshd

$ fail2ban-client status               # fail2ban: что мониторится
$ fail2ban-client status sshd           # jail sshd: забаненные IP
$ fail2ban-client set sshd unbanip 1.2.3.4
ulimit / limits.conf
Лимиты ресурсов на процесс/сессию. Самый частый — nofile (max open files). Дефолтные 1024 ломают БД, веб-сервера и контейнеры под нагрузкой.
Примеры
$ ulimit -a                            # все лимиты текущей сессии
$ ulimit -n                            # max open files
$ ulimit -n 65536                      # поднять (только до hard limit)
$ cat /proc/1234/limits                # лимиты конкретного процесса
  # для постоянных лимитов — /etc/security/limits.conf:
  # deploy  soft  nofile  65536
  # deploy  hard  nofile  65536
  # для systemd — LimitNOFILE=65536 в [Service]
⚠️ Перед iptables -P INPUT DROP или ufw enable на удалённом сервере — ВСЕГДА сначала разреши SSH! Иначе залочишься. На облачных VM держи в карман emergency-доступ через web-console / serial.

Performance & I/O
iostat, iotop, sar, dstat, pidstat, atop — глубокий анализ
iostat / iotop
Анализ дисковой подсистемы. iostat — статистика по устройствам, ключевая метрика %util (близко к 100 = диск перегружен) и await (latency запросов). iotop — кто из процессов грузит диск (как top, но про I/O).
Примеры
$ iostat -xz 1                       # расширенная статистика, обновление 1с
$ iostat -xz 1 5                     # 5 замеров и выйти
$ iotop -oP                          # только активные процессы, без потоков
$ iotop -b -n 3                       # batch для скриптов
  # iostat ключевые колонки:
  # r/s, w/s    — IOPS чтения/записи
  # rkB/s, wkB/s — пропускная способность
  # await       — средняя latency запроса (мс)
  # %util       — % времени, когда диск был занят
  # aqu-sz      — средняя длина очереди
sar (sysstat)
Главный инструмент исторического анализа. Демон sysstat непрерывно собирает метрики (CPU, mem, disk, net) и сохраняет в /var/log/sa/. Можно посмотреть, что творилось на сервере вчера в 03:00.
Примеры
$ sar -u 1 5                          # CPU сейчас
$ sar -r 1 5                          # память
$ sar -d 1 5                          # диски
$ sar -n DEV 1 5                       # сеть
$ sar -u -f /var/log/sa/sa15             # CPU 15-го числа
$ sar -q -s 03:00:00 -e 04:00:00        # LA с 3 до 4 утра
$ sar -B                              # paging (swap-активность)
pidstat / dstat / atop
pidstat — статистика по конкретным процессам (CPU, mem, I/O, page faults) с историей. dstat — комбинированный live-вывод. atop — как top, но сохраняет историю (post-mortem анализ инцидентов).
Примеры
$ pidstat 1                            # CPU всех процессов
$ pidstat -r 1                         # память + page faults
$ pidstat -d 1                         # дисковая активность
$ pidstat -p 1234 1                    # конкретный PID
$ pidstat -t -p 1234 1                 # по потокам
$ dstat -tcmdn                       # time, cpu, mem, disk, net в одной таблице
$ atop                                # live
$ atop -r /var/log/atop/atop_20260513    # история за вчера
Netflix 60-sec perf checklist
Канонический набор для первой минуты на новом сервере с проблемой. От Brendan Gregg (Netflix Performance Engineering).
Запускай по очереди
$ uptime                              # 1. LA: растёт или стабилен?
$ dmesg | tail                        # 2. ядро: OOM? драйверы?
$ vmstat 1                            # 3. CPU/mem/swap общая картина
$ mpstat -P ALL 1                      # 4. CPU per-core (один загружен?)
$ pidstat 1                           # 5. топ-процессы CPU
$ iostat -xz 1                        # 6. диски: util, await
$ free -m                              # 7. память
$ sar -n DEV 1                          # 8. сеть: throughput
$ sar -n TCP,ETCP 1                     # 9. TCP errors, ретрансмиты
$ top                                 # 10. живая картина в конце
🎯 USE method (Brendan Gregg) для каждого ресурса проверяй: Utilization (загружен %), Saturation (есть очередь?), Errors (есть ошибки?). Ресурсы: CPU, mem, disk, net, потоки, sockets. Метод покрывает 80% инцидентов.

Production Runbook
Чёткие алгоритмы для типовых инцидентов — то, что любят спрашивать senior'ы
🔥 Инцидент: сервер тормозит
Подключись и не паникуй: ssh на проблемный хост, проверь w / uptime. Запиши LA — будешь сравнивать.
Категоризируй ботлнек: vmstat 1. Высокий us → CPU. Высокий wa → I/O. Высокий si/so → swap. Растёт r (run queue) → перегрузка.
Найди виновника: по CPU — top/pidstat 1; по I/O — iotop -oP; по памяти — ps aux --sort=-%mem | head.
Проверь зависимости: сетевые сервисы — ss -s; БД — есть ли блокировки/долгие запросы; внешние API — curl -w latency.
Смотри логи приложения и системы: journalctl -p err --since "10 min ago", dmesg -T | tail -50, логи приложения.
Краткосрочная стабилизация: рестарт, скейл-аут, rollback деплоя, drain трафика. НЕ начинай дебажить root cause в острой фазе — стабилизация → расследование.
Постмортем: сохрани логи, atop-историю, метрики. Напиши blameless RCA: что было, что сделали, что улучшим.
💾 Инцидент: "диск переполнен"
Подтверди: df -h — какой раздел. Если 100% — приложение уже падает с ENOSPC.
Проверь inode'ы: df -i. Бывает, что байтов полно, а inode'ы кончились (миллионы мелких сессионных файлов).
Найди где: du -h --max-depth=1 / 2>/dev/null | sort -rh | head. Спускайся вглубь.
Удалённые открытые файлы: если du и df расходятся — lsof | grep deleted | sort -k7 -rn | head. Нужен рестарт процесса (или : > /proc/PID/fd/N освободит сейчас).
Бэкап перед удалением: сожми и сдвинь то, что точно мусор, но удалить страшно: gzip /var/log/big.log.
Долгосрочно: настрой logrotate, добавь алерты на 80% диска, mount-poins по сервисам (логи на отдельный диск).
🐌 Инцидент: "приложение медленно отвечает"
Воспроизведи: curl -w "@curl-fmt.txt" -o /dev/null -s https://api/health — увидишь TCP/TLS/TTFB latency отдельно.
Локализуй слой: с LB → app slow? с app → DB slow? с app → external slow? Если у тебя есть APM (DataDog/NewRelic/Sentry) — смотри traces.
Проверь pool'ы: connection pool на БД — не исчерпан ли? ss -s — много ли соединений? Воркеры приложения — все заняты?
База данных: SELECT * FROM pg_stat_activity WHERE state <> 'idle', долгие запросы, locks, vacuum.
Сеть: mtr до зависимостей, tcpdump на ретрансмиты, sar -n TCP,ETCP 1.
Ресурсы хоста: CPU saturated? GC pauses? Swap? Iowait?
Что изменилось: релиз, конфиг, миграция, рост трафика? journalctl --since со старта инцидента.
🔒 Инцидент: "не могу подключиться по SSH"
С другого хоста проверь, жив ли хост: ping, nc -zv host 22.
Verbose SSH: ssh -vvv user@host — увидишь на каком этапе обрыв (DNS, TCP, ключи, auth).
Right key? Проверь ~/.ssh/config и -i. ssh-add -l — какие ключи в агенте.
Если есть доступ к хосту иначе (cloud console, IPMI): journalctl -u sshd -f, sshd -t (валидность конфига), fail2ban-client status sshd (не забанен ли IP).
Firewall: iptables -L -n на хосте + Security Groups в облаке.
Лимит сессий: MaxSessions, MaxStartups в sshd_config; ulimit -n; нет ли исчерпания PAM.
📦 Инцидент: "контейнер постоянно рестартует"
Логи: docker logs --tail 200 -f <id> или kubectl logs --previous <pod> для упавшего контейнера.
Exit code: docker inspect <id> --format '{{.State.ExitCode}}'. 137 = OOM-kill, 139 = segfault, 143 = SIGTERM.
OOM-kill: dmesg -T | grep -i "killed process"; в k8s — kubectl describe pod, ищи OOMKilled. Решение: поднять memory.limit или починить утечку.
Healthcheck/probes: liveness/readiness падают слишком рано? Слишком жёсткие timeout? Приложение медленно стартует.
Зависимости: БД, секреты, конфиги — доступны при старте? Часто CrashLoopBackOff = init не находит DSN.
Запусти руками: docker run --entrypoint sh -it <image> или kubectl debug — посмотри что внутри, проверь права, env.

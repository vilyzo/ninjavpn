📘 Мануал: Диагностика TCP и UDP подключений

Полный справочник по проверке доступности сервисов, трассировке маршрутов и диагностике сетевых проблем с использованием стандартных и продвинутых утилит.

Оглавление: 1. Базовые утилиты (ping, traceroute, mtr) | 2. Проверка TCP | 3. Проверка UDP | 4. Windows-альтернативы | 5. Чеклист диагностики

1. Базовые утилиты: какие протоколы они используют?

УтилитаПо умолчаниюМожет использовать TCP?Примечание
pingICMP Echo❌ Нет (стандарт)Работает на сетевом уровне. Не использует порты.
traceroute (Linux)UDP 33434+✅ Да (-T / -P tcp)Windows tracert использует ICMP.
mtrICMP / UDP✅ Да (в некоторых форках)Комбинирует ping + traceroute. IPv6 → ICMPv6.
⚠️ Почему не TCP по умолчанию?
TCP требует трёхэтапного рукопожатия, что добавляет задержку и может давать ложные результаты при закрытых портах. ICMP/UDP созданы именно для диагностики связности без установки сессии.

2. Проверка TCP-подключений

🔹 Netcat (nc)

# Быстрая проверка порта
nc -vz example.com 443

# С отправкой данных (HTTP-запрос)
printf "HEAD / HTTP/1.1\r\nHost: example.com\r\n\r\n" | nc example.com 80

🔹 Telnet

telnet example.com 22
# Успешное подключение → порт открыт
# "Connection refused" → порт закрыт
# Таймаут → возможная блокировка фаерволом

🔹 curl (для HTTP/HTTPS)

curl -v --connect-timeout 5 https://example.com:443
# Вывод покажет этапы TCP handshake, TLS negotiation и ответ сервера

🔹 tcping (ICMP-альтернатива для портов)

tcping example.com 443
# Работает как ping, но измеряет задержку до TCP-порта
💡 Совет: Если ICMP заблокирован, но сервис работает — используйте nc -vz или tcping вместо обычного пинга.

3. Проверка UDP-подключений

⚠️ Важно: UDP — протокол без установки соединения. «Успешная» отправка пакета не гарантирует доставку или обработку на стороне приложения.

🔹 Netcat (nc -u)

# Клиент: отправить пакет
echo "probe" | nc -u -w2 example.com 53

# Сервер: слушать порт (на другой машине)
nc -u -l -p 52000

🔹 nmap (сканирование UDP)

# Конкретный порт
nmap -sU -p 53,123,161 192.168.1.1

# Статусы:
# open         → порт отвечает
# open|filtered → нет ответа (возможно, открыт или отфильтрован)
# closed       → получен ICMP port unreachable

🔹 iperf3 (тест пропускной способности)

# Сервер
iperf3 -s

# Клиент (UDP-тест)
iperf3 -u -c 192.168.1.100 -b 10M -t 10

🔹 Продвинутая диагностика

# nping (из пакета nmap)
nping --udp -p 53 --data-string "test" 8.8.8.8

# hping3
hping3 --udp -p 123 -d 100 192.168.1.1

# Мониторинг трафика (на принимающей стороне)
sudo tcpdump -n udp port 52000

5. Чеклист диагностики проблем

# 1. Базовая связность
ping example.com

# 2. Маршрут
traceroute example.com   # Linux
tracert example.com      # Windows

# 3. DNS
nslookup example.com
dig example.com

# 4. TCP-порт
nc -vz -w 3 example.com 443

# 5. UDP-порт + мониторинг
# На приёмнике:
sudo tcpdump -n udp port 52000
# На отправителе:
echo "test" | nc -u -w1 example.com 52000
🔑 Ключевые параметры nc:
-v — подробный вывод | -z — режим сканирования | -u — UDP | -w N — таймаут (сек) | -l — прослушивание

Полезные комбинации

# TCP: проверить 3 порта подряд
for port in 80 443 8080; do
  nc -vz -w2 example.com $port
done

# UDP: отправить пакет и ждать ответ 2 сек
echo "probe" | nc -u -w2 example.com 123
